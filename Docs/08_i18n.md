## 0. 決定事項（サマリ）

| 論点 | 決定 | 理由 |
|---|---|---|
| 台本の作り方 | **日本語正本 → 行保存翻訳**。ネイティブ生成（最初から外国語で台本生成）は作らない | 日本語の無いプロジェクトは現時点で想定しない（ユーザー方針）。行構造（line_id）維持が Aロールパネル・素材紐付け・話者配役をそのまま継承する唯一の方式 |
| 保存構成 | `episodes/epNN/locales/{lang}/` に言語別成果物のみ。footage/・a_roll/ は原語と共有 | MINOR互換。既存コンテナは locales/ を無視して動く |
| TTSエンジン | **言語→エンジン自動選択**: 原語(ja)=irodori（現状維持）、他言語=OmniVoice。統一しない・毎回選択させない | irodori の日本語品質は実績済みで捨てる理由がない。OmniVoice の日本語品質は未検証。自動選択なら「問題なく動く」を満たす（上書き設定は残す） |
| OmniVoice の起動 | 新コンテナ **omnivoice-tts-server (:8089→コンテナ内8880, profiles: gpu)** を常時起動。**vendor採用**: maemreyo/omnivoice-server を install.ps1 で clone（irodoriと同じ流儀）、無改造で使用 | maemreyo版はMIT・81コミット・адapterパターンで上流API変更に強い設計。フルスクラッチ自作よりバグが少ない。**Whisper自動書き起こし経路には絶対に触れない**運用（ref_text必須化）で唯一の既知バグ(torchcodec不整合)を無害化できることをスパイクで実証済み |
| GPU 4GB の排他（**2026-07-08 Phase1実機検証で再訂正**） | **両モデル同時ロードは実際に成立する**（実測: irodori(bf16,約2.4GB)+OmniVoice(fp16,約1.9GB)を同時ロードして両方`/health`健全＝合計使用3859MB／空き104MB）。ただし空きがほぼゼロなので**「両方に同時に生成リクエストを送る」（並行推論）はOOMリスクが高い**（未検証・Phase3で確認）。設計は「起動・待機は共存可、**同時推論だけ避ける**」に緩和。tts-agent側は各生成リクエストの前後で相手側が推論中でないことだけ確認すればよく、モデルのロード/アンロード制御自体は不要 | 想定していた「同時ロード不可＝どちらか一方しか起動できない」という前提が実測で覆った。OmniVoiceは常時起動のまま（§0上段）で運用でき、ユーザー操作（restart等）を要求する場面を大幅に減らせる |
| OmniVoice のロード方式（**2026-07-08訂正**） | vendorのFastAPI lifespanがコンテナ**起動時に無条件でモデルをeagerロード**する（irodoriのような遅延ロードではない）。トグルなし（upstream仕様） | `docker compose --profile gpu up -d` で一括起動すると、omnivoiceは常に約1.9GBをすぐ確保する。irodoriは要求時まで0GB。VRAM見積もりはこの非対称を前提にする |

## 1. データフロー（全体像）

```
rough_script.txt（日本語）
  → scripting: script.json（日本語・正本・承認済み）
      → tts(ja=irodori) / footage / a_roll / edit   ← 現行どおり
      → 【新】translate(lang) → locales/{lang}/script.json（行保存翻訳）
           → 【新】tts(lang=omnivoice) → locales/{lang}/tts.json + audio/
           → 【新】edit(lang) → locales/{lang}/edit/（OTIO+SRT）
           ※ footage / a_roll は原語のものを共有（line_id/section が一致するため）
```

- 翻訳は**確定 script.json のみ**を入力にする（draft は対象外）。
- 原語 script.json を修正したら翻訳は「古い」扱い（§3 の `source_script_hash` で判定・UIで再翻訳を促す）。

## 2. フォルダ構成

```
episodes/ep01/
├── script.json / tts.json / audio/ / footage/ / a_roll/ / edit/   ← 原語（現状のまま）
└── locales/
    └── en/
        ├── script.json    ← 行保存翻訳（line_id/order/speaker_id/section/emotion は原本と同一）
        ├── tts.json
        ├── audio/         ← line_XXX.wav
        └── edit/          ← timeline.otio / subtitles.srt / edit.json
```

## 3. スキーマ変更（DATA_SCHEMA.md 2.1.0 で正式化）

### 3a. project.json（MINOR: episodes[].locales と config.tts.locale_engines を追加）

```json
{
  "language": "ja",                          // 既存フィールドを「原語（正本言語）」として正式に意味付け
  "episodes": [{
    "number": 1,
    "status": { "...現行どおり（原語の工程）..." },
    "locales": {
      "en": {
        "title": "English episode title",    // 翻訳された話タイトル（publish用・翻訳時にLLMが生成）
        "status": {
          "translation": "done",             // 行保存翻訳
          "tts": "pending",
          "editing": "not_started",
          "video_edit": "not_started"
        }
      }
    }
  }],
  "config": {
    "tts": {
      "engine": "irodori",                   // 原語エンジン（現状維持・後方互換）
      "locale_engines": { "default": "omnivoice" }  // 任意。lang→engine の上書きマップ
    }
  }
}
```

- locale の status に `footage` は無い（原語と共有のため）。
- エンジン解決規則（tts-agent）: `lang == project.language` → `config.tts.engine` ／ それ以外 → `locale_engines[lang]` → `locale_engines.default` → 組み込み既定 `"omnivoice"`。

### 3b. locales/{lang}/script.json（原語 script.json と同スキーマ＋鮮度メタ）

```json
{
  "schema_version": "1.0.0",
  "project_id": "...",
  "locale": "en",
  "translated_from": "ja",
  "source_script_hash": "sha256(原語script.jsonのlines textの連結)",
  "lines": [ /* 原本と同一line_id・textのみ英語 */ ],
  "sections": [ /* 原本コピー */ ],
  "metadata": { /* 原本準拠 */ }
}
```

- **翻訳しないフィールド**: `notes`（制作メモ）、`speaker_name`（表示キャッシュ・従来どおり都度解決）。
- `speed` / `pause_after_sec` は原本値をコピー（言語別調整は行編集UIで可能＝既存機構がそのまま効く）。

### 3c. 声カタログ: OmniVoice用 ref クリップ + ref_text サイドカー（新規・重要・実装済み）

```
shared/voices/irodori/
├── KUJO-OK.mp3           ← irodori用の元音声（そのまま）
├── KUJO-OK.ref.wav       ← ★新設: OmniVoice用の短尺(3-10秒)参照クリップ。文の区切りで切り出す
└── KUJO-OK.ref.txt       ← ★新設: .ref.wav の正確な書き起こし（UTF-8・原語＝日本語）
```

- OmniVoice のゼロショットクローンは `ref_audio + ref_text`（正確な transcript）で品質が決まる。**ref_audio は irodori 用フル尺ではなく専用の短尺クリップ**を使う（OmniVoice公式推奨が3-10秒・長いと劣化）。
- **ref_text 無しでの OmniVoice 生成は 409 でブロック**する。理由（スパイク実測）: ref_text 省略 → サーバーが Whisper を CUDA へ自動ロードし**常駐** → VRAM枯渇で全生成が10倍劣化。
- **実装済み（2026-07-08）**: 既存3声（AOI-OK/KUJO-OK/LUKA-OK）について、各 `.mp3` から文の区切りで4.8〜5.9秒の `.ref.wav` を切り出し、Cloudflare Whisper（research-agent が既に使っている `cloudflare_text.transcribe` と同じAPI。**JSON+base64形式が正**、生バイナリ`application/octet-stream`では `Invalid input` エラーになる点に注意＝今回で判明した新事実）で正確な日本語書き起こしを取得し `.ref.txt` に保存済み。
- 新規キャラ追加時のワークフロー: 声を登録する際に運用側（人力 or 今後のUI拡張）で3-10秒の参照クリップを用意し、Whisperまたは手動で `.ref.txt` を書く。**手順を自動化するUIは今回のスコープ外**（Phase 1 は既存3声の手当てのみ）。
- character.json は**変更なし**（`voice` 単一のまま）。他言語も同じ参照音声でクロスリンガルクローンする（スパイクで実証済み）。言語ごとに声を替えたい要望が出たら §2b 既定路線の `voices: [{engine, lang, voice_id}]` 配列化で対応（今回はスコープ外）。

### 3d. 冒頭ハルシネーション＋語尾切れ問題 — 根本原因と対策（2026-07-08 Phase1で解決）

**症状**: 正確な ref_text にしても、生成音声の**冒頭に本来のテキストに無い幻覚フレーズ**（例: ES生成で「¡Wow! ¡Henryo Masou!」という実在しない語句、ユーザー報告では「韓国語っぽい音声」）が混入し、さらに**文末が欠落**する現象が発生した。

**調査手法**: 生成音声を人間の耳ではなく **Cloudflare Whisperで書き起こし、意図したテキストと機械的に突き合わせる**ことで客観的に検証（人間の音質確認を待たずに再現・特定できた）。

**根本原因**: `omnivoice` パッケージの `generate()` の既定値 `position_temperature=5.0`（マスク位置選択の温度＝非決定的サンプリング）が、クロスリンガル（日本語参照音声→英語/スペイン語生成）時に冒頭の幻覚フレーズを生みやすい。この幻覚が事前推定された出力長（テキストベースの `RuleDurationEstimator`）の時間予算を消費し、結果として実際の文末が生成時間内に収まらず欠落する。**2つの症状は同一原因の表裏**だった。

**対策（確定・実装反映必須）**: クローン生成リクエストには常に **`position_temperature=0`（+ `class_temperature=0`、既定で0なので明示は任意）を指定する**。

実測比較（KUJO声・同一ref_text、Whisper書き起こしで target と完全一致するか）:

| 設定 | ES書き起こし結果 |
|---|---|
| 既定（position_temperature=5.0） | 「¡Wow! ¡Henryo Masou! Hola a todos. ...」— 冒頭に幻覚句混入 |
| `position_temperature=0` | 「Hola a todos. Bienvenidos de nuevo al canal. Hoy vamos a hablar de las últimas noticias en inteligencia artificial. Espero que disfruten este episodio.」— **target と完全一致** |

英語でも同様に `position_temperature=0` で target 完全一致を確認済み（速度もRTF≈0.7で実用圏、劣化なし）。

**tts-agent実装への反映**: `engines/omnivoice.py`（§4）の clone リクエストは `position_temperature=0, class_temperature=0` を**固定デフォルト**として送る（ユーザー調整不要・多様性より正確性を優先）。

## 4. TTSエンジン抽象化（tts-agent）

現状: `tts_client.py` が irodori 直結の単一実装。`"irodori"` リテラルが routes.py 4箇所＋キャッシュキー `"irodori-stereo"` に散在。

改修:
```
tts-agent/app/core/engines/
├── __init__.py        ← resolve_engine(pj, lang) → エンジン実装を返すレジストリ
├── irodori.py         ← 現 tts_client.py の移設（挙動不変）
└── omnivoice.py       ← /v1/audio/speech/clone (multipart) を叩く。ref_audio=キャラ声ファイル実体、
                          ref_text=サイドカー必須。モノラル→ステレオ化は audio_utils を共用
```

- 全TTSルート（単行/全行/プレビュー）に `lang` クエリを追加。**省略時 = 原語 = 現行動作**（後方互換）。
- パス解決は `project_manager` の `get_script_path()` / `get_audio_dir()`（+tts.json パス）に `lang` 引数を追加し locales/ へ分岐（変更箇所を1ファイルに集約）。
- キャッシュキーの engine 名はハードコードをやめ実エンジン名を使う（テキスト自体が言語で異なるため衝突はないが、正確を期す）。
- tts.json には従来どおり `engine` を記録（locale版は `"omnivoice"` になる）。

### 4b. GPU 排他ガード（4GBの現実解・2026-07-08実機検証で全面簡素化）

**実測（Phase1）**: irodori と OmniVoice は**両方同時ロードした状態で共存できる**（3859MB使用／104MB空き。両`/health`とも健全）。当初想定した「片方しかロードできない」は誤りだった。よって tts-agent 側にロード/アンロード制御・コンテナ再起動指示・409ガードは**不要**。

残るリスクは1点だけ: 空きVRAMがほぼゼロのため、**両エンジンに同時並行で推論リクエストを送ると OOM の可能性がある**（未検証・Phase3で要確認）。対策は軽量:

| 対応 | 内容 |
|---|---|
| 並行実行の抑止 | tts-agent プロセス内に**プロセス全体で1個のグローバル推論ロック**（asyncio.Lock）を持たせ、irodori/OmniVoiceどちらの生成もこのロックを取ってから実行。「全行生成」のようなループ処理も直列実行のまま（現状も`max_concurrent_synthesis:1`等ですでに直列的なので追加コストは小さい） |
| 保険 | Phase3実装後に「原語生成中に別言語の全行生成を実行」のような実際の並行シナリオで一度だけ実測し、OOMが出るなら上記ロックで解決 / 出ないならロックも不要と判断 |

- `.env` の逃げ道は維持: OmniVoiceを `OMNIVOICE_DEVICE=cpu` にすればVRAM衝突は原理的にゼロ（RTF≈5・バッチ生成なら実用圏）。
- コンテナに docker socket は渡さない（許可リスト方針: コンテナからのホスト操作は不可）— この方針は維持するが、上記の理由によりPhase1想定ほど頼る場面がなくなった。

## 5. 翻訳エンドポイント（scripting-agent）

```
POST /projects/{id}/episodes/{n}/translate
body: { "lang": "en", "model": "(任意・既定は既存LLMルーティング)", "instructions": "(任意・訳調指示)" }
```

- 入力: 確定 `script.json`（無ければ 409）。既存翻訳があれば `?force=true` でのみ上書き（インポートと同じ流儀）。
- LLM: 既存 `llm_client`（Gemini無料枠→OpenRouter無料）。プロンプトは「話し言葉として自然に・原文と同程度の長さ・行番号と行数を厳守」。
- **バリデーション必須**（speaker-id-normalization の教訓）: 出力行数が原本と一致しない場合はチャンク分割リトライ→それでも不一致なら error。line_id はLLM出力を信用せず**順序で原本から機械的に再割当**。
- 併せて話タイトルの翻訳も生成し `episodes[].locales[lang].title` へ。
- 完了時 `episodes[].locales[lang].status.translation = done`、`tts = pending`。

## 7. OmniVoice サーバー統合（新コンテナ omnivoice-tts-server :8089、実装済み・2026-07-08）

**方針転換（設計時から変更）**: 自作フルスクラッチではなく **maemreyo/omnivoice-server を vendor 採用**（irodori-tts-serverと全く同じ流儀＝`install.ps1` が `--depth 1` clone、無改造でビルド）。理由: MIT・81コミット・adapterパターンで上流`omnivoice`パッケージのAPI変更を1ファイルに隔離する設計になっており、フルスクラッチ自作よりバグ面で有利。**唯一の既知バグ（torchcodec/ffmpeg ABI不整合）は Whisper自動書き起こし経路でのみ発火し、ref_text必須運用で経路ごと回避できる**ことをスパイクで実証済み（詳細下記）。

| 項目 | 仕様 |
|---|---|
| ソース | `tts-agent/omnivoice-tts-server-src`（install.ps1が clone。gitignore対象） |
| イメージ | 同梱の `Dockerfile.cuda` を無改造で使用（`nvidia/cuda:12.8.0-cudnn-runtime-ubuntu22.04`ベース） |
| モデル | `k2-fsa/OmniVoice`（Apache 2.0）。HFキャッシュは `./tts-agent/omnivoice-models:/home/ubuntu/.cache/huggingface`（bind・gitignore） |
| ポート | ホスト8089 → コンテナ8880（アプリのハードコード値。envの`OMNIVOICE_PORT`は現状無視される＝upstreamの既知の癖） |
| API | `POST /v1/audio/speech/clone`（multipart: text, ref_audio, ref_text**必須運用**, speed, num_step）／`GET /health`（`model_loaded`公開） |
| ロード方針 | **コンテナ起動時に無条件でeagerロード**（irodoriのような遅延ロードではない。vendorのFastAPI lifespanにハードコード・トグル無し）。**アイドル自動アンロードは無し**（§0のGPU排他表を参照。`--profile gpu up -d`一括起動時は常時約1.9GB確保する点に注意） |
| Whisper | サーバー自体には未組込。ref_text省略時のみ**upstream `omnivoice` パッケージ内部**がWhisperを動的ロード→常駐。**tts-agent/呼び出し側がref_textを絶対に省略しない**運用で完全回避 |
| デバイス | `OMNIVOICE_DEVICE=cuda`（既定）/ `cpu` 退避可（`.env`不要・compose environment直書き） |
| compose | `profiles: ["gpu"]`。**override(docker-compose.override.yml)には追記不要**（irodori-tts-serverも同様に無し＝GPU本番系サービスはdevホットリロード対象外という既存パターンに合わせた） |

### スパイク実測値（2026-07-08, RTX 3050 Ti Laptop 4GB）
- モデル常駐 ≈1.9GB（システム込み使用 2.4GB／空き1.5GB）。ロード時間 ≈3.5分（HFキャッシュ済みなら短縮見込み）。
- EN: 8.5秒音声を5.5秒で生成（RTF≈0.65）／ ES: 15.1秒音声を6.9秒で生成（RTF≈0.46）。
- Whisper 常駐時は同じ生成が 65〜89秒に劣化（10倍以上）→ ref_text 必須化の根拠。
- **重要な追加事実（Phase 1実装時に判明）**: ref_audioは model.generate(ref_audio=<path>, ref_text=...) へファイルパスで直接渡され、この経路は torchcodec を使わず soundfile ベースで安定動作する。torchcodec エラーは **ref_text 省略時の自動書き起こし（Whisper）専用の内部経路でのみ**発火する。

## 9. 罠チェックリスト（本件で踏みやすい既知の罠）

- 台本UIの**二重実装**（director/scripting 両方に反映）
- compose は **base+override の2枚**、新コンテナのマウント追加も2枚とも
- `.env` にキー追加後は `--force-recreate`（再起動だけでは未反映）
- Windows curl に日本語JSONを直接渡さない（`--data @file`）
- OmniVoice に ref_text 無しリクエストを絶対に通さない（Whisper常駐＝実質サービス停止）
- ポート 8089 を新規使用（8001-8006/8088/8188 と衝突しないこと・CLAUDE.md サービス地図の更新）
